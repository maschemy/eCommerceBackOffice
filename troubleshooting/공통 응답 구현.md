# 🛠️ Trouble Shooting: API 공통 응답 전역 처리 (ResponseBodyAdvice)

---

## 📌 배경 (Background)
* **상황:** 공통 응답 클래스를 구현한 후, 모든 컨트롤러의 `return` 타입을 일일이 수정해야 하는 번거로움 발생.
* **이슈:** 반복적인 코드 수정의 비효율성을 해소하고자 자동화 방식 도입 결정.
* **기존 방식의 문제점:**
    * **코드 중복:** 모든 엔드포인트마다 응답을 감싸는 보일러플레이트 코드 반복.
    * **관심사 혼재:** 컨트롤러가 비즈니스 로직 외에 '응답 포맷팅' 책임을 가짐.
    * **유지보수 저하:** 응답 규격 변경 시 모든 컨트롤러 코드 수정 필요.

---

## 🧐 ResponseBodyAdvice란?
* **정의:** `@ResponseBody`가 사용된 컨트롤러에서 반환된 값을 응답 본문(Response Body)에 쓰기 전에, 사용자 지정 형식으로 가공하거나 변환할 수 있게 해주는 인터페이스입니다.

---

## 💡 해결 방안: 전역 처리기 도입
* **방식:** Spring의 `@RestControllerAdvice`와 `ResponseBodyAdvice`를 구현하여, 컨트롤러가 반환한 데이터를 전달하기 직전에 가로채어 공통 포맷으로 자동 래핑하는 `CommonResponseHandler`를 도입.

### 도입 후 장점
* **비즈니스 로직 집중:** 컨트롤러는 DTO 등 순수 데이터만 반환하면 되므로 가독성 향상.
* **일관성 보장:** 전역 처리기에서 자동으로 규격을 맞춰주어 휴먼 에러 방지.
* **단일 유지보수점:** 응답 규격 변경 시 한 곳만 수정하면 됨.

---

## ⚠️ 구현 중 문제 및 해결 (Issue & Solution)

1. **String 타입 캐스팅 에러**
    * **현상:** 컨트롤러에서 문자열(`String`) 반환 시 `ClassCastException` 발생.
    * **해결:** `@ExceptionHandler`를 통해 예외 처리된 결과가 공통 응답 형식으로 변환되는 과정에서 타입 불일치가 발생함. 예외 발생 시에는 공통 응답 형식을 적용하지 않도록 전역 예외 처리 로직 수정.

2. **기타 발생 이슈**
    * **외부 라이브러리 간섭:** Spring Actuator 등 외부 응답까지 래핑되어 문서가 깨지는 현상.
    * **중복 래핑:** 이미 `CommonResponse`로 반환된 데이터가 한 번 더 감싸지는 현상.

---

## 📉 단점 및 대안
* **단점:** 성공/실패 시나리오에 따라 응답 결과가 달라져 `supports` 메서드 구현이 복잡해질 수 있음.
* **향후 대안:** * `ResponseBodyAdvice`를 여러 개 등록하여 성공/실패 케이스별로 분리 구현.
    * 프로젝트 규모가 커지고 예외 케이스가 복잡해질 경우, 전역 처리 대신 **명시적인 정적 팩토리 메서드**(`return ApiResponse.ok(data);`)를 사용하여 코드 추적성을 높이는 방안 고려.
