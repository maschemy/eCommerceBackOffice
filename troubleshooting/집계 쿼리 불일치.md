# 🛠️ Trouble Shooting: 도메인별 삭제 전략 차이로 인한 집계 쿼리 불일치

---

## 📌 배경 (Background)
* **상황:** 대시보드 기능을 개발하면서 전체 관리자 수, 전체 고객 수, 전체 주문 수 등 도메인별 통계 데이터를 집계해야 하는 상황 발생.
* **이슈:** 단순히 `count()` 쿼리만 사용하면 될 것이라 생각했으나, 도메인별로 **삭제 전략(Soft Delete, Hard Delete, 상태 관리 등)**이 달라 동일한 방식으로 집계가 불가능함을 인지.

---

## 🧐 문제 원인 (Root Cause)
팀 내 도메인 설계 시 삭제 전략이 통일되지 않아 집계 시 적용해야 할 필터링 조건이 제각각이었습니다.

| 도메인 | 삭제 전략 | 집계 시 주의사항 |
| :--- | :--- | :--- |
| **Admin** | Soft Delete | `deletedAt IS NULL` (탈퇴 관리자 제외) |
| **Customer** | Soft Delete | `deletedAt IS NULL` (탈퇴 고객 제외) |
| **Product** | Hard Delete | 별도 필터링 없이 `count()` 가능 |
| **Order** | 상태값 관리 | `status != CANCEL` (취소 주문 제외) |

---

## 💡 해결 방안: 도메인별 집계 메서드 분리
각 도메인별 삭제 전략을 분석하여, Repository에 적합한 집계 조건을 명시적으로 추가했습니다.

```java
// 도메인별 상황에 맞게 쿼리 분리
adminRepository.countByDeletedAtIsNull();         // Soft Delete — 탈퇴 관리자 제외
customerRepository.countByDeletedAtIsNull();      // Soft Delete — 탈퇴 고객 제외
productRepository.count();                        // Hard Delete — 그대로 집계
orderRepository.sumTotalRevenue(OrderStatus.CANCEL); // 취소 주문 제외
```

---

## 📝 결말 및 배운 점 (Lesson Learned)
* **결과:** 도메인별 전략에 최적화된 집계 메서드를 구현함으로써 정확한 통계 데이터 확보.
* **배운 점:** * 잘못된 집계 조건 사용은 데이터 정합성 문제(삭제/취소된 데이터 포함)로 직결됨을 확인.
    * 팀 단위 개발 시 **삭제 전략(Soft vs Hard Delete)을 초기에 통일하거나 문서화**하는 것이 유지보수와 데이터 정합성 관리에 매우 중요함을 체감.
